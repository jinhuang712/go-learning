# 5. 深浅拷贝 (Copy vs Clone) 的底层逻辑

在 Java 中，如果你把一个对象赋值给另一个变量，默认只是传递了**引用（浅拷贝）**。如果你想完整复制一个对象，你需要实现 `Cloneable` 接口并重写 `clone()` 方法（深拷贝）。

但在 Go 中，情况完全不同，其核心法则只有一条：
**Go 语言中的所有赋值操作 (`=`)、函数参数传递，全都是“值传递（浅拷贝）”！**

这听起来很简单，但当遇到复合类型时，很容易让人产生错觉。我们需要根据类型来分类讨论。

## 5.1 基础类型与结构体的拷贝 (完全的深拷贝)
对于 `int`、`float`、`bool` 以及**内部不包含指针的结构体 (Struct)**，赋值操作会把内存中的数据完完整整地复制一份。修改新变量，绝对不会影响老变量。

```go
type Point struct { X, Y int }
p1 := Point{1, 2}
p2 := p1 // 发生完整的内存拷贝 (深拷贝)
p2.X = 99 
// p1.X 依然是 1
```

## 5.2 引用类型的拷贝 (浅拷贝陷阱)
Go 中的引用类型主要有三种：**切片 (Slice)**、**映射 (Map)**、**通道 (Channel)**。
当你把一个 Slice 赋值给另一个变量时，Go 依然是“值传递”，但它传递的是什么？是 Slice 的**底层结构体**（包含 Pointer, len, cap）。

因为复制过去的 Pointer 依然指向同一块底层数组，这就导致了**浅拷贝 (Shallow Copy)** 的效果：
```go
s1 := []int{1, 2, 3}
s2 := s1 // 只拷贝了 slice 结构体(24字节)，底层的数组没有被拷贝！

s2[0] = 999 
// s1[0] 也会变成 999！因为它们共享底层数组。
```

## 5.3 结构体内部包含引用类型 (半深半浅)
这是业务开发中最容易出 Bug 的地方。如果你的结构体里包含了一个 Slice 或 Map：
```go
type User struct {
    Name string
    Tags []string // 这是一个切片 (引用类型)
}

u1 := User{Name: "Alice", Tags: []string{"admin", "vip"}}
u2 := u1 // 拷贝了整个结构体
```
此时：
- `u2.Name = "Bob"`：不会影响 `u1`，因为 `string` 是不可变的值类型。
- `u2.Tags[0] = "guest"`：**会影响 `u1`！** 因为 `Tags` 这个切片被浅拷贝了，`u1` 和 `u2` 的 `Tags` 共享同一个底层数组。

## 5.4 如何在 Go 中实现真正的深拷贝 (Deep Copy)？

Go 标准库**没有**提供类似 `clone()` 的通用深拷贝函数（因为反射深拷贝性能极差）。如果你需要深拷贝一个包含引用类型的复杂结构体，你有两种选择：

1. **手动逐个字段赋值 (推荐，性能最好)**:
   ```go
   func (u *User) Clone() *User {
       // 1. 拷贝值类型的字段
       clone := &User{Name: u.Name}
       // 2. 手动深拷贝切片字段
       if u.Tags != nil {
           clone.Tags = make([]string, len(u.Tags))
           copy(clone.Tags, u.Tags) // 使用内置的 copy 函数
       }
       return clone
   }
   ```
2. **通过 JSON 序列化反序列化 (最懒，性能极差)**:
   把对象 `json.Marshal` 变成字节流，再 `json.Unmarshal` 到一个新对象里。在对性能要求苛刻的微服务中，**强烈不推荐**这种做法。
