# 2. 切片 (Slice) 核心剖析与坑点

切片是 Go 中最常用的集合类型。你可以把它当成 Java 的 `ArrayList`，但它的底层实现更有意思。

## 2.1 切片的底层结构（非常重要）
切片在底层其实是一个只有 3 个字段的小结构体（在 64 位机器上占 24 字节）：
1. **Pointer (指针)**: 指向底层的实际数组。
2. **Length (长度, len)**: 当前切片中实际元素的个数。
3. **Capacity (容量, cap)**: 从指针位置开始，底层数组能提供的最大连续空间。

```go
// 源码中的 slice 结构类似这样
type slice struct {
    array unsafe.Pointer
    len   int
    cap   int
}
```

## 2.2 创建方式：字面量 vs make
### 字面量创建 (极其容易与数组混淆)
在 Go 中，通过字面量创建切片和数组的语法极其相似，**唯一的区别就在于中括号里有没有数字**：
- `arr := [3]int{1, 2, 3}`：这是**数组**（中括号有具体数字）。
- `arr := [...]int{1, 2, 3}`：这是**数组**（让编译器数数字）。
- `slice := []int{1, 2, 3}`：这是**切片**（中括号是**空**的！）。

当执行 `[]int{1, 2, 3}` 时，Go 底层其实偷偷做了两件事：
1. 在内存中分配了一个隐藏的数组 `[3]int{1, 2, 3}`。
2. 创建了一个切片结构体，把它的指针指向那个隐藏数组，并设置 `len=3, cap=3`。

### 使用 make 创建切片
除了字面量，在工程中动态创建切片最常用的方式是使用内置的 `make` 函数：
- `s := make([]int, 5)`：创建一个长度 `len=5`，容量 `cap=5` 的切片。此时里面已经填充了 5 个 `0`，你可以直接 `s[0] = 10`。
- `s := make([]int, 0, 10)`：**大厂最推荐的写法**。创建一个长度 `len=0` 的空切片，但底层预先分配了 `cap=10` 的连续内存。这样你在后续 `append` 时，前 10 次都不会触发性能昂贵的内存重新分配。

## 2.3 切片的截取 (Slicing) 与副作用
当你使用 `subSlice := nums[1:3]` 截取时：
- `subSlice` 并没有复制数据，它只是创建了一个新的 `slice` 结构体（新的指针、新的 len、新的 cap）。
- **坑点**: `subSlice` 和原切片 `nums` **共享同一个底层数组**！修改 `subSlice[0]` 会直接改变原数组的数据。

## 2.4 如何安全地 Clone (克隆) 一个切片？
如果你不希望截取出的切片和原数组产生任何关联，必须进行深拷贝 (Deep Copy)：

1. **使用 `copy` 内置函数**:
   ```go
   sub := nums[1:3] 
   clone := make([]int, len(sub)) 
   copy(clone, sub) 
   ```
2. **使用 `append` 技巧** (推荐):
   ```go
   // []int(nil) 代表一个空切片。
   // nums[1:3]... 意思是把这个切片里的元素“打散”追加进去。
   // 因为前面的 nil 切片容量为 0，这会强迫 Go 重新分配一块干净的内存！
   clone := append([]int(nil), nums[1:3]...) 
   ```

**❌ 致命错误示范：千万不能反着写！**
```go
// 错误：试图把空切片追加到有剩余容量的切片上
wrongClone := append(nums[1:3], []int(nil)...)
```
**原因**：`append` 的底层逻辑是“只要被追加方的 Capacity 还够，就不分配新内存”。`nums[1:3]` 底层仍指向那个大数组，容量很充足；而你追加了 0 个元素。结果 `append` 根本没有触发扩容，直接把原共享切片返回了！修改 `wrongClone` 依然会污染原数组。

## 2.5 Append 与扩容机制
在官方文档里，`append` 的签名是：`func append(slice []Type, elems ...Type) []Type`
- 这里的 `elems ...Type` 是可变参数（类似 Java 的 `Type... args`），代表你要追加的一个或多个值。
- 如果当前 `len < cap`，直接在底层数组的对应位置写入，`len++`。
- **如果当前 `len == cap`，触发扩容**：
  1. Go 会在内存中开辟一块更大的新数组（通常容量翻倍，或者按一定策略平滑增长）。
  2. 将老数组的数据拷贝过去，再把新元素追加进去。
  3. 返回一个指向**新数组**的切片。

### 为什么必须写成 `slice = append(slice, v)`？
既然 `slice` 内部包含了指针，为什么我不能像 Java 一样只写 `append(slice, v)` 呢？
**因为 `slice` 结构体本身是按“值传递”传给 `append` 函数的！**
当 `append` 触发扩容时，它在内部创建了一个**指向新数组的全新 slice 结构体**。如果你不把这个新的结构体通过 `=` 重新赋值给外面的 `slice` 变量，外面的变量依然傻傻地指向那个老数组，而且它的 `len` 也没有增加！
