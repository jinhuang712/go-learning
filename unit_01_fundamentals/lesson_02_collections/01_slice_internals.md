# 1. 数组 (Array) 与 切片 (Slice) 核心剖析

在 Go 语言中，理解数组和切片的本质区别，是避开无数潜在 Bug 的关键。

## 1.1 数组 (Array)：值类型、定长
- **定义**: `var arr [3]int`。**长度是类型的一部分**，`[3]int` 和 `[4]int` 是两种完全不同的类型。
- **传递方式**: **值传递**。如果在函数间传递数组，Go 会把整个数组完整拷贝一份（如果数组很大，性能开销极高）。
- **在 Java 中的对比**: Java 的数组（如 `int[]`）其实是对象引用，传递的是地址；而 Go 的数组是彻头彻尾的“值”。

## 1.2 切片 (Slice)：引用类型、动态扩容
切片是 Go 中最常用的集合类型。你可以把它当成 Java 的 `ArrayList`，但它的底层实现更有意思。

### 切片的底层结构（非常重要）
切片在底层其实是一个只有 3 个字段的小结构体（在 64 位机器上占 24 字节）：
1. **Pointer (指针)**: 指向底层的实际数组。
2. **Length (长度, len)**: 当前切片中实际元素的个数。
3. **Capacity (容量, cap)**: 从指针位置开始，底层数组能提供的最大空间。

```go
// 源码中的 slice 结构类似这样
type slice struct {
    array unsafe.Pointer
    len   int
    cap   int
}
```

### 切片的截取 (Slicing) 与副作用
当你使用 `subSlice := nums[1:3]` 时：
- `subSlice` 并没有复制数据，它只是创建了一个新的 `slice` 结构体（新的指针、新的 len、新的 cap）。
- **坑点**: `subSlice` 和原切片 `nums` **共享同一个底层数组**！修改 `subSlice[0]` 会直接改变原数组的数据。

### Append 与扩容机制 (Capacity)
当你使用 `append(slice, value)` 追加元素时：
- 如果当前 `len < cap`，直接在底层数组的对应位置写入，`len++`。
- **如果当前 `len == cap`，触发扩容**：
  1. Go 会在内存中开辟一块更大的新数组（通常容量翻倍，或者按一定策略增长）。
  2. 将老数组的数据拷贝过去。
  3. 将新元素追加进去。
  4. 返回一个指向**新数组**的切片。
- **神坑**: 扩容后，原切片和新切片**不再共享底层数组**！这也是为什么我们必须写 `slice = append(slice, v)` 来接收返回值的原因。
